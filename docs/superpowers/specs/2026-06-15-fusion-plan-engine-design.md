# `/fusion` — многомодельный движок планирования

- **Дата:** 2026-06-15
- **Статус:** дизайн утверждён, готов к плану реализации
- **Тип:** Claude Code skill / plugin (мета-тулинг, отдельный репозиторий `~/code/fusion`)

## Проблема и цель

Нужен инструмент, который делает не «нормальный», а максимально проработанный план: с исследованием проблематики, перебором решений, ограничениями, сравнением, спайками рисковых допущений и интервью оператора там, где нужно взвешенное человеческое решение. На готовый план — пост-проход «можно ли проще/лучше/меньше».

Тезис, на котором стоит инструмент, — «fusion beats frontier» (OpenRouter): ансамбль из нескольких **разных семейств** моделей обгоняет одну топовую, потому что у разных лайнеджей разные слепые зоны. Применяем это к фазе планирования: Claude, Codex/GPT и DeepSeek-R1 независимо проектируют и затем дебатируют.

Выход — план-документ. Реализация плана — вне скоупа (отдельный beast-forge / forge / `improve execute`).

## Не-цели (YAGNI)

- Роутер «выбрать одну модель» — это не fusion.
- Отдельная модель-судья — судит Claude-оркестратор, он уже запущен.
- Персистентная БД дебатов — артефакты лежат файлами рядом с планом.
- Авто-применение плана к коду — fusion только планирует.
- Backlog-lifecycle (`reconcile`, `--issues`) — возможное v2, не в первой версии.

## Контекст: чем отличается от shadcn/improve

improve и fusion режут «много моделей» по разным осям и **дополняют** друг друга:

| | Ось разнообразия | Модели на планировании |
|---|---|---|
| **improve** | вертикальная (цена): умная планирует → дешёвые исполняют | субагенты одного семейства (всё Claude) |
| **fusion** | горизонтальная (семейство): Claude + GPT + DeepSeek дебатируют сам план | три разных лайнеджа |

У improve N параллельных субагентов делят слепые зоны одной модели — третье семейство закрывает именно эту дыру. «Умный план → дешёвое исполнение» из improve ортогонально и относится к фазе реализации, которая здесь за скобками.

**Берём у improve как есть (он зрелее по качеству планов, MIT):**

1. **Recon глотает intent-доки** (ADR, PRD, `CONTEXT.md`, `DESIGN.md`) — решённые трейдоффы не переоткрываются как находки.
2. **Vet-шаг:** оркестратор сам перечитывает каждый процитированный `file:line` перед синтезом. Совпадает с правилом анти-амплификации (`verify-claims.mjs`); для fusion обязательно — внешние модели галлюцинируют пути чаще Claude-субагентов.
3. **Формат исполняемого плана:** self-contained, verification-gates с ожидаемым выводом, hard STOP-условия, штамп git-коммита + drift-check.

**Чем fusion уникален:** кросс-семейные дебаты на планировании, спайки рисковых допущений (throwaway worktree), интервью оператора на взвешенных развилках.

## Архитектура: пайплайн

```
0. Frame      Claude собирает контекст (repo / codegraph / GBrain / intent-доки) → общий бриф всем
1. Research   3 модели ║ независимо: проблематика, решения, ограничения → черновой план каждая
2. Debate     3 модели ║ видят 2 чужих плана → критика (миражи, дыры, скрытая сложность) + правят свой
3. Spike      оркестратор берёт самые рискованные допущения → throwaway-спайки в worktree → факты назад
4. Interview  на РЕАЛЬНЫХ развилках → AskUserQuestion оператору
5. Synthesize Claude vet → сводит лучшее из трёх + спайки + решения → пишет план-документ
6. Better/Simpler  3 модели атакуют готовый план «проще? меньше? лучше?» → финальная правка
```

Маппинг на роли beast-forge (researcher → skeptic → critic → simplifier), но разнообразие из трёх семейств, а не N Claude-субагентов.

### Адаптивная глубина

Оркестратор сам решает, сколько раундов жечь, по степени расхождения моделей:

- Черновики фазы 1 сошлись (согласны по подходу, рискам, шагам) → ранний стоп после vet+synthesize, как `improve quick`.
- Расходятся непримиримо → доп. раунд дебатов и/или спайк спорного допущения.
- `--depth lite|full` — ручной оверрайд (full = минимум 2 раунда + обязательный спайк + обязательный пост-пасс).

Критерий «сошлись/разошлись» оценивает Claude-оркестратор по финальным черновикам, не по самооценке участников.

## Транспорт и адаптеры (заземлено)

Все три участника на планировании работают **read-only** (читают репо, выдают текст-план):

| Участник | Вызов | Заметки |
|---|---|---|
| Claude | Claude Code субагент (Agent tool) для независимого черновика; главная сессия — оркестратор | read-only в фазах 1–2 |
| Codex/GPT | `codex exec "<prompt>"` в read-only sandbox | точный флаг sandbox подтверждается на спайке |
| DeepSeek-R1 | `opencode run --dir <repo> -m openrouter/deepseek/deepseek-r1 --agent <ohmyopenagent> "<prompt>"` | агент `oh-my-openagent` (`~/.config/opencode/oh-my-openagent.json`); точное имя из `opencode agent list` |

Подтверждённое на этапе дизайна: `codex exec [PROMPT]` неинтерактивен; `opencode run` поддерживает `-m provider/model`, `--agent`, `--dir`, `--format json`; DeepSeek доступен как `openrouter/deepseek/deepseek-r1` (+ `deepseek-chat-v3.1` и нативные варианты).

Ключ OpenRouter для DeepSeek — через `op-agent` в рантайме, не инлайн в скрипте.

## Изоляция и worktree

- Фазы планирования (1, 2, 6) — read-only, без записи в репо.
- Фаза 3 (спайки) — каждый спайк в отдельном `git worktree` от HEAD, throwaway. Cleanup через `trap` — worktree не утекает даже при падении.
- Только Claude-оркестратор пишет финальный артефакт (план-док + транскрипт).

## Интервью оператора

Claude паузит и зовёт `AskUserQuestion` **только** на реальных развилках:

- модели расходятся непримиримо и нет объективного критерия;
- необратимое или дорогое решение;
- бизнес/продуктовый выбор, требующий человека.

Не дёргать оператора по мелочам — по умолчанию решает сам.

## Формат плана (взято у improve)

План пишется для слабейшего правдоподобного исполнителя:

- **Self-contained:** точные пути, выдержки текущего кода, конвенции репо с файлом-образцом, проверенные команды. Никаких «как обсуждалось выше».
- **Verification-gates:** каждый шаг кончается командой и ожидаемым выводом; «готово» машинно-проверяемо.
- **Hard boundaries:** явный out-of-scope и STOP-условия («если X — стоп и доложи»).
- **Drift-check:** штамп git-коммита, против которого писался план.

## Дисциплина верификации (анти-амплификация)

Перед синтезом (фаза 5) оркестратор механически сверяет каждую цитату/число/`file:line`, на которых строится вывод, против источника. Пересказ ≠ цитата. Находка без подтверждения — выкинуть или пометить «claimed, unverified». Вывод, ведущий к необратимому акту, не строится на одной модели.

## Артефакты и layout

Репозиторий `~/code/fusion` зеркалит shadcn/improve (ставится так же):

```
~/code/fusion/
  .claude-plugin/           манифест плагина
  skills/fusion/
    SKILL.md                playbook фаз, парсинг /fusion <task> [--depth lite|full] [--rounds N]
    fusion.sh               fan <model> <promptfile> | spike (worktree lifecycle+cleanup) | collect
  docs/superpowers/specs/   этот документ
```

Выход одного запуска:

```
docs/superpowers/specs/YYYY-MM-DD-<topic>-plan.md      финальный план (improve-формат)
docs/superpowers/specs/YYYY-MM-DD-<topic>-debate.md    транскрипт: кто что предлагал, чем проиграл
```

`SKILL.md` — playbook (фазы, промпты, судейство глубины, vet, синтез). `fusion.sh` — единственный долгоживущий отлаживаемый артефакт (lifecycle worktree, параллельный запуск CLI, захват логов/диффов). Логика оркестрации детерминированная — в скрипте, не переизобретается каждый запуск.

## Режимы отказа

- CLI упал/таймаут → участник выбывает, дебаты идут с оставшимися (минимум 1). Логируем честно, не подделываем ответ.
- Таймаут на участника (5–10 мин) против зависаний.
- Спайк упал → допущение помечается «не подтверждено», план это явно отражает.
- Cleanup worktree через `trap` — гарантирован при любом выходе.
- Нет ключа OpenRouter → DeepSeek выбывает с явным предупреждением, fusion продолжает на двух.

## Самое рискованное допущение → спайк перед сборкой

**Неизвестность:** надёжно ли `codex exec` и `opencode run` гоняются неинтерактивно — авто-аппрув без зависания на запросе разрешения, захват вывода, приемлемая латентность/стоимость трёх моделей × раунды.

**Спайк (2 минуты):** на тривиальной задаче («спланируй добавление healthcheck-эндпоинта») прогнать обе внешние CLI по одному разу в read-only, убедиться, что возвращают текст без зависания на пермишене и вывод парсится. Снимает главный риск до написания оркестрации.

## Открытые вопросы

Нет блокирующих. Точные флаги read-only sandbox для `codex exec` и имя агента opencode уточняются механически на спайке (фаза реализации), не требуют решения оператора.
